# Learning Graph PoC — Build Instructions for Claude Code

## Goal

A Tauri desktop app (Rust backend, Svelte frontend) that renders a DAG of
learning nodes. Users can generate/edit the graph, "zoom in" on a node to
spawn a nested subgraph, and verify their understanding via an LLM. Both the
database and the LLM provider must be swappable (BYO-DB, BYO-AI) behind
abstract Rust traits, decided at runtime via user config, not compile-time
feature flags.

Build this in phases, in order. Do not skip ahead to the frontend before the
backend traits and a working mock implementation compile and pass tests.

---

## Phase 1 — Core data model

Define in `src-tauri/src/model.rs`:

```rust
use serde::{Deserialize, Serialize};
use uuid::Uuid;
use chrono::{DateTime, Utc};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Graph {
    pub id: Uuid,
    pub title: String,
    /// None for a root/top-level graph. Some((node_id)) if this graph is a
    /// "zoom in" subgraph nested under a node in a parent graph.
    pub parent_node_id: Option<Uuid>,
    pub created_at: DateTime<Utc>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Node {
    pub id: Uuid,
    pub graph_id: Uuid,
    pub title: String,
    pub description: String,
    pub status: MasteryStatus,
    /// IDs of nodes in the SAME graph that must be mastered first.
    pub prerequisite_ids: Vec<Uuid>,
    /// Set once a subgraph has been generated by zooming into this node.
    pub subgraph_id: Option<Uuid>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type")]
pub enum MasteryStatus {
    Unknown,
    Learning,
    Mastered { verified_at: DateTime<Utc> },
}
```

Notes for Claude Code:
- `prerequisite_ids` living on `Node` (adjacency list) is sufficient for a
  PoC — no separate edges table needed.
- A "zoom in" does NOT delete the parent node. It creates a new `Graph` row
  with `parent_node_id` set, and sets `subgraph_id` on the parent node. This
  is what makes the depth-based fog rendering possible later: depth is just
  "how many `parent_node_id` hops from the graph currently in focus."

## Phase 2 — Database trait (BYO-DB)

Define in `src-tauri/src/db/mod.rs`:

```rust
use crate::model::{Graph, Node};
use async_trait::async_trait;
use uuid::Uuid;
use anyhow::Result;

#[async_trait]
pub trait GraphStore: Send + Sync {
    async fn create_graph(&self, graph: &Graph) -> Result<()>;
    async fn get_graph(&self, id: Uuid) -> Result<Option<Graph>>;
    async fn delete_graph(&self, id: Uuid) -> Result<()>;

    async fn create_node(&self, node: &Node) -> Result<()>;
    async fn get_node(&self, id: Uuid) -> Result<Option<Node>>;
    async fn list_nodes(&self, graph_id: Uuid) -> Result<Vec<Node>>;
    async fn update_node(&self, node: &Node) -> Result<()>;
    async fn delete_node(&self, id: Uuid) -> Result<()>;

    /// Re-link children when a node with children is deleted — see notes.
    async fn reparent_children(&self, deleted_node: Uuid, new_prereqs: Vec<Uuid>) -> Result<()>;
}
```

Implementations (both using `sqlx`, same trait):
- `SqliteStore` — local file, default for BYO-local.
- `PostgresStore` — remote/cloud, used for both self-hosted Postgres and our
  managed cloud tier.

Since both back ends use `sqlx`, prefer writing the SQL once against
`sqlx::Any` if pooling allows it; otherwise keep two small structs that both
implement `GraphStore` — do not let SQL dialect differences leak into
`model.rs` or the Tauri command layer.

Deletion behavior to implement explicitly (don't leave implicit): deleting a
node with children must either (a) promote the deleted node's own
prerequisites to become the children's new prerequisites, or (b) orphan the
children with no prerequisite (user's choice via a flag on the delete
command). Never silently drop the reference.

Provider selection: a `StoreConfig` enum (`Sqlite { path }` /
`Postgres { url }`) read from app config at startup, producing a
`Box<dyn GraphStore>` behind an `Arc` in Tauri's managed state.

## Phase 3 — LLM trait (BYO-AI)

Define in `src-tauri/src/llm/mod.rs`. Normalize the request/response shape —
do not leak Ollama's or Anthropic's native API schema into this trait.

```rust
use async_trait::async_trait;
use serde::{Deserialize, Serialize};
use anyhow::Result;

/// A generated node + its prerequisite titles (resolved to IDs after insert).
#[derive(Debug, Deserialize, Serialize)]
pub struct GeneratedNode {
    pub title: String,
    pub description: String,
    pub prerequisite_titles: Vec<String>,
}

#[derive(Debug, Deserialize, Serialize)]
pub struct GeneratedGraph {
    pub title: String,
    pub nodes: Vec<GeneratedNode>,
}

#[derive(Debug, Serialize)]
pub struct VerificationQuestion {
    pub prompt: String,
    /// e.g. "requires computed numeric answer", "requires code", "requires derivation"
    pub expects: String,
}

#[derive(Debug, Deserialize)]
pub struct VerificationResult {
    pub passed: bool,
    pub feedback: String,
    /// If the model thinks a prerequisite gap caused the failure, it can
    /// propose new nodes here — this feeds the Remediation flow.
    pub suggested_new_prereqs: Vec<GeneratedNode>,
}

#[async_trait]
pub trait LlmProvider: Send + Sync {
    /// Generate a full starter graph for an arbitrary goal string.
    async fn generate_graph(&self, goal: &str) -> Result<GeneratedGraph>;

    /// "Zoom in": generate a subgraph for one existing node, given its
    /// title/description as context so the subgraph fits coherently under it.
    async fn zoom_node(&self, node_title: &str, node_description: &str) -> Result<GeneratedGraph>;

    /// Pose a verification question for a node.
    async fn generate_verification_question(&self, node_title: &str, node_description: &str) -> Result<VerificationQuestion>;

    /// Grade the user's answer against the rubric implied by the question.
    async fn grade_answer(&self, question: &VerificationQuestion, user_answer: &str) -> Result<VerificationResult>;
}
```

Implementations:
- `OllamaProvider` — calls local Ollama HTTP API (`/api/generate` or
  `/api/chat`), model name configurable.
- `AnthropicProvider` — calls the Anthropic Messages API, using the
  structured-JSON-output pattern (system prompt demands JSON only, no
  preamble) to fill `GeneratedGraph` / `VerificationResult`.

Important for verification quality (this is the product's actual
differentiator — do not let it degrade silently): ship ONE fixed rubric
system prompt used identically regardless of provider. The provider only
changes fluency/capability, never the grading criteria. Also surface a
`capability_hint: Low | Medium | High` on `LlmProvider` (hardcoded per
provider/model) so the frontend can show a confidence indicator next to
"Mastered" statuses graded by a weaker local model.

Provider selection: same pattern as `StoreConfig` — an `LlmConfig` enum
(`Ollama { host, model }` / `Anthropic { api_key, model }`) producing a
`Box<dyn LlmProvider>`.

## Phase 4 — Tauri commands

Thin wrappers in `src-tauri/src/commands.rs` that call the trait objects held
in managed state. One command per trait method, roughly:

```
get_graph, list_nodes, create_node, update_node_status, delete_node,
generate_starter_graph, zoom_into_node, request_verification, submit_answer
```

Keep business logic (re-parenting on delete, splicing a generated subgraph
back into the parent graph's edges) in a separate `src-tauri/src/service.rs`
module, not inline in the command handlers — commands should just
deserialize args, call the service layer, serialize the result.

## Phase 5 — Frontend rendering (Svelte, no WebGL/canvas libs)

Structure: each `Graph` renders as an absolutely-positioned stack of layers
inside one perspective container.

```css
.graph-viewport {
  perspective: 1200px;
  perspective-origin: center;
}

.graph-layer {
  position: absolute;
  inset: 0;
  transition: transform 400ms ease, filter 400ms ease, opacity 400ms ease;
  transform-style: preserve-3d;
}

.depth-0 {           /* currently focused graph */
  transform: translateZ(0) scale(1);
  filter: none;
  opacity: 1;
  pointer-events: auto;
}

.depth-1 {            /* one level below current focus */
  transform: translateZ(-300px) scale(0.85);
  filter: blur(3px) opacity(0.5);
  pointer-events: none;
}

.depth-2 {            /* two levels below */
  transform: translateZ(-600px) scale(0.72);
  filter: blur(6px) opacity(0.2);
  pointer-events: none;
}

.depth-hidden {
  display: none;       /* anything past depth-2: don't render at all */
}
```

Behavior:
- "Zoom in" on a node = navigate focus to its `subgraph_id`. Animate by
  re-assigning depth classes (old depth-0 becomes depth-1's z/blur/scale
  target, its child subgraph animates from depth-1 to depth-0) and let CSS
  transitions handle the motion. Do not literally move the DOM node between
  containers mid-animation — swap the class, let the transition interpolate.
- "Zoom out" is the reverse: depth-0 animates back out to depth-1 position
  and the previous depth-1 (its parent graph) animates in to depth-0.
- Depth is computed from the chain of `parent_node_id` lookups from whatever
  graph is currently focused, capped at 2 for rendering purposes per the
  fog spec — don't bother fetching graphs beyond depth 2, no reason to pay
  the DB round-trip for something that isn't rendered.
- Node status (Unknown/Learning/Mastered) drives color, not depth — depth
  and mastery are orthogonal and shouldn't be conflated in the CSS.

Use Svelte stores for the focused-graph-id and a derived store for the
depth-annotated stack of graphs to render, so depth-class assignment is
computed once and consumed by a simple `{#each}` rather than recomputed in
markup.

## Suggested build order for Claude Code sessions

1. `model.rs` + in-memory mock `GraphStore` + mock `LlmProvider` (hardcoded
   fixture responses) — get the Tauri commands and Svelte shell wired end to
   end against mocks before touching real DB/LLM code. This isolates
   frontend bugs from backend/provider bugs.
2. `SqliteStore` real implementation, swap in for the mock, verify CRUD via
   the already-working frontend.
3. `AnthropicProvider` real implementation for `generate_graph` only —
   verify the structured-JSON parsing round-trips into `GeneratedGraph`
   before wiring `zoom_node` and verification, since JSON-shape bugs are
   easiest to debug with the simplest call first.
4. Wire `zoom_node` end to end, including the graph-splicing service logic.
5. Wire verification + the fog/depth rendering last — it's the most
   frontend-heavy phase and benefits from having real generated data to test
   against rather than fixtures.
6. `PostgresStore` and `OllamaProvider` last — these are "prove the
   abstraction actually works" steps, not new features.
